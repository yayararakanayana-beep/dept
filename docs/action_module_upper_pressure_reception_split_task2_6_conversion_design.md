# 作用モジュール 上位圧受容分離 RC1 / Task 2-6

日本語名: **圧-作用変換関数の設計/実装**

コードラベル: `ActionModule_UpperPressureReceptionSplit_RC1_Task2_6_PressureActionConverter`

## 0. 位置づけ

Task 2-6では、Task 2-5a〜2-5fで作った検証表を材料として、`PressureIntentBundle` または同等の圧入力を、作用候補へ変換する関数を作る。

ただし、ここで作るものは **作用強度補正器** ではない。

圧の強さに対して結果が不足していても、作用強度をブーストして結果を無理に圧へ合わせない。

## 1. 固定方針

Task 2-6の固定方針は次である。

```text
圧を、意図に合う作用候補へ変換する。
ただし、作用強度を補正して結果を無理に圧へ合わせない。
```

例:

```text
圧が 5
結果が 5

圧が 10
結果が 7
```

この場合、後者は失敗としてブーストするのではなく、

```text
圧10に対して、この作用経路では7程度しか届かない
```

という情報として残す。

したがって、次を禁止する。

```text
圧10 → 強度補正 → 結果10
```

## 2. Task 2-6がやること

```text
圧入力を読む
既存の作用→圧対応表を見る
圧に合う作用候補を作る
状態・相互作用で危険な候補を監査する
変換不足を情報として残す
```

## 3. Task 2-6がやらないこと

```text
作用強度を補正して結果を圧に合わせる
結果不足を埋めるために作用をブーストする
最終Actionを決める
ActionFrameを作る
作用モジュールを呼ぶ
世界を実行する
下位レイヤー状態を勝手に判定する
```

## 4. 変換の基本形

Task 2-6の基本形は次である。

```text
圧
↓
作用→圧対応表を逆向きに使う
↓
作用候補
↓
状態窓口による候補監査
↓
相互作用監査
↓
作用候補表
```

ここで出力されるのは、最終Actionではない。

出力は、同時発火門や後続監査へ渡すための候補表である。

## 5. 作用強度の扱い

Task 2-6では、作用強度曲線による補正を適用しない。

Task 2-5eの作用強度曲線は、Task 2-6では参照可能な監査情報に留める。

Task 2-6の出力では、次を分けて残す。

```text
pressure_signal:
  圧と基本対応率から出た素の候補信号。

expected_result_signal:
  その候補が検証上どの程度の結果へ届きそうかを示す参考値。

pressure_result_gap:
  圧に対して結果が足りない差分。
```

この `pressure_result_gap` を埋めるために、Task 2-6内で作用を強めてはいけない。

## 6. 状態と相互作用の扱い

状態・相互作用は、作用強度補正ではなく、候補監査として使う。

つまり、次のように扱う。

```text
状態が危険:
  候補を保留または禁止として印をつける
  ただし強度をブーストしない

相互作用が危険:
  候補ペアに危険印をつける
  必要なら片方を保留または禁止候補として印をつける
  ただし強度をブーストしない
```

Task 2-6では、Task 2-5bの `boost` は強度上げとして使わない。

`boost` は「候補として残しやすい」という意味に限定し、数値強度には反映しない。

## 7. 出力に必ず残す情報

Task 2-6の出力表には、最低限次を残す。

```text
元の圧成分
圧の方向
圧の強さ
候補作用
基本対応率
素の候補信号
期待結果信号
圧-結果差分
状態監査結果
相互作用監査結果
強度補正未使用フラグ
圧に合わせるブースト未使用フラグ
最終Actionではないフラグ
ActionFrame未作成フラグ
作用モジュール未呼び出しフラグ
```

## 8. 完了条件

Task 2-6は、次を満たしたとき完了とする。

```text
圧入力から作用候補表が作れる。
作用強度補正が入っていない。
圧と結果の差分が残る。
圧不足をブーストで埋めていない。
状態・相互作用は候補監査に限定されている。
最終Actionを決めていない。
ActionFrameを作っていない。
作用モジュールを呼んでいない。
```
