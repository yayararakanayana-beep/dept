# 汎用関係場 G2 固定5軸接続層

## 1. 到達点

G2は、現在の固定5軸 `G_t`・`K_t` とRF-1〜RF-10を変更せず、固定構造を汎用関係場側へ渡す接続層を実装する。

今回実装したもの。

1. 固定5軸構造プロファイル
2. RF-2格子成果物から汎用構造成果物への接続層
3. 軸・セル・辺・面・境界の識別子付き台帳
4. 接続演算子と「境界の境界が0」の構造検証器
5. 固定5軸 `K_t` の因果的接頭部を可変集合表示へ変換する読み取り専用接続層
6. 関係場の量を識別子付きレコードとして保存する成果物基盤
7. 決定論的保存、SHA-256マニフェスト、上書き禁止

## 2. 今回固定しているもの

固定値の正本は、既存の凍結済みG/K・RF契約である。次の専用ファイルは、どの正本から何を受け取るかだけを定義する。

```text
configs/fixed5axis_relation_structure_g2.json
```

この対応表には、軸名、区分、セル数、辺数、面数を重複記載しない。接続層は凍結済み契約から読み、相互不一致があれば停止する。

汎用中核の契約は次である。

```text
configs/generic_relation_field_g2_contract.json
```

汎用検証器は、軸数・セル数・辺数・面数を入力台帳から導出する。固定5軸の件数を判定式へ直接埋め込まない。

## 3. 固定の意味

固定するのは構造識別である。

- 軸
- 区分
- セル
- 辺
- 面
- 境界集合

次は固定しない。

- `G_t` の質量
- 辺流れ
- 流路幅
- 結合強度
- 回復可能性
- 新規駆動
- 未解決残差
- リスク構造

## 4. 構造成果物

固定5軸接続層は次を保存する。

```text
contract.json
structure_profile.json
axis_registry.json
cell_registry.npz
edge_registry.npz
face_registry.npz
boundary_registry.json
boundary_membership.npz
field_record_schema.json
validation.json
manifest.json
```

配列位置は保存上の順序であり、意味上の識別子ではない。

識別規則。

```text
axis: 軸台帳の axis_id
cell: cell/<旧RF-2セルID>
edge: edge/<旧RF-2辺ID>
face: face/<旧RF-6面ID>
boundary: <axis_id>/lower または <axis_id>/upper
```

## 5. 既存RFとの互換境界

G2は次を実測比較する。

- RF-2のセル順序
- RF-2の辺順序・起点・終点・軸
- RF-2の接続演算子
- RF-3の辺輸送費用
- RF-6の面順序・面を構成する辺・向き
- `B_1 B_2 = 0`

RF-3〜RF-10の科学量の完全比較はG3で行う。G2は、それらが共通して参照できる構造・識別・成果物基盤を作る段階である。

## 6. 因果的G/K接続

予測起点`t`を指定した場合、接続層は次だけを表示する。

```text
G_0, G_1, ..., G_t
```

保存形式は、各フレームとセル台帳をオフセットで対応づける可変集合形式である。

```text
identity.json
frame_registry.jsonl
mass_records.npz
provenance.json
validation.json
manifest.json
```

接続層は未来の質量値、未来の台帳行、正解ファイルを使用しない。元の `G_t`・`K_t` へ書き戻さない。

## 7. 関係場レコード

関係場の量は、固定長ベクトルを正本とせず、次の識別子付きレコードとして保存する。

```text
record_id
feature_id
feature_group
scope_type
scope_ids
value
lower_bound
upper_bound
unit
normalization_id
availability_status
evidence_status
source_refs
```

G2で許可する証拠状態は観測と候補推定だけである。未来予測を関係場観測へ混ぜない。

## 8. 実行例

まず既存RF-2成果物を生成する。

```bash
python scripts/relation_field_grid_rc1.py build \
  --output artifacts/relation_field_grid_rc1
```

固定5軸構造を汎用形式へ接続する。

```bash
python scripts/generic_relation_field_g2.py build-fixed5-structure \
  --grid-artifact artifacts/relation_field_grid_rc1 \
  --output artifacts/generic_relation_structure_g2
```

検証する。

```bash
python scripts/generic_relation_field_g2.py validate-structure \
  --input artifacts/generic_relation_structure_g2
```

因果的なG/K接頭部を表示する。

```bash
python scripts/generic_relation_field_g2.py build-history-view \
  --trajectory <固定5軸G/K軌道ディレクトリ> \
  --structure artifacts/generic_relation_structure_g2 \
  --to-t 8 \
  --output artifacts/generic_relation_history_view_t8
```

## 9. 停止条件

次の場合は補正せず停止する。

- G/K、RF-1、RF-2の軸契約不一致
- 構造ハッシュ不一致
- 接続演算子の起点・終点・符号不一致
- 面境界の不整合
- `K_t` 接頭部の不連続
- G/K来歴と構造台帳の不一致
- 負質量または質量非保存
- 未知の軸・セル・辺・面識別子
- 成果物改ざん
- 観測成果物への未来予測証拠の混入

## 10. G2で実装していないもの

- 汎用生成 `G_t`・`K_t`
- 時間途中の構造変更
- 構造間対応
- RF-3〜RF-10の全計算の汎用版
- ミクロ予測器
- マクロゲーム構造予測器
- 分布幾何シミュレーター
- 予測精度評価
- 不可逆性の正式推定
- 作用評価・作用選択
- Task 3 Rev1との統合

未知構造で同じ精度が出るという主張は行わない。

## 11. 次のG3

G3では、固定5軸入力に対して既存RF-3〜RF-10と新しい汎用経路の科学量を比較する。

同時に、3軸・7軸、区分数違い、面なし、穴あり、複数連結成分、識別子順序変更などの合成構造で、汎用中核に固定5軸依存が残っていないかを検証する。

合成構造試験は汎用生成G/Kや未知構造での予測精度を保証するものではない。

設定値・挙動ごとに正本を一つにする。

| 対象 | 正本 |
| --- | --- |
| G/K軸・区分・形状・セル数 | `configs/fixed5axis_gk_rc1_contract.json` |
| RF-2セル・辺・距離・接続演算子 | `configs/relation_field_grid_rc1_contract.json` |
| RF-3辺輸送費用 | `configs/relation_field_single_transition_rc1_contract.json` |
| RF-6面数・面構造 | `configs/relation_field_hodge_decomposition_rc1.json` |

## 14. 実装順序

1. G2汎用契約と固定5軸対応表を検証する
2. 既存G/K・RF-1・RF-2契約の軸定義が一致することを確認する
3. 既存RF-2成果物を独立検証する
4. 軸・セル・辺・面・境界の識別子付き台帳へ変換する
5. 接続演算子と面境界を再計算する
6. 既存RF-2の識別・接続、RF-3の輸送費用、RF-6の面と比較する
7. 構造ハッシュとマニフェストを確定して原子的に保存する
8. 指定時点までのG/K接頭部を、元成果物を変更せず可変集合表示へ変換する
9. 関係場量を識別子付きレコードとして保存・再検証する
10. 正試験・否定試験・既存RF回帰試験を実行する

## 15. 重要要件契約

### 15.1 固定値の隔離

- 要件: 汎用中核が固定5軸の軸名・件数を知らない
- 必須実装: 固定値は既存凍結契約から読み、固定5軸対応表は参照関係だけを持つ
- 禁止する近道: コードやG2汎用契約へ5軸、3,125セル、12,500辺、20,000面を再記載する
- 検証: テストがG2汎用契約・実装・対応表の固定値文字列を監査し、0軸3セル2辺の合成構造を同じ検証器へ通す

### 15.2 既存構造との科学的互換

- 要件: 新構造成果物が既存RF-2/RF-3/RF-6と同じ構造を表す
- 必須実装: セル、辺、接続符号、輸送費用、面辺、面向きを実データから比較する
- 禁止する近道: 件数一致や`true`の保存だけで互換と判定する
- 検証: 既存RF-2検証器とRF-6面生成器を実行し、全配列を要素単位で比較する。接続符号を反転した否定試験を失敗させる

### 15.3 因果的G/K接頭部

- 要件: 時点`t`の表示が `G_0:t` だけを含む
- 必須実装: 台帳を`t`で停止し、選択行の質量だけを読み、ハッシュ鎖を接頭部内で再計算する
- 禁止する近道: 全履歴を読み込んでから未来行を削る、未来正解を読む、元G/Kへ書き戻す
- 検証: 4時点の正本から`t=1`を作り、2フレーム・6,250質量レコードだけであること、元成果物の全ファイルハッシュが不変であることを確認する。存在しない時点と来歴不一致を拒否する

### 15.4 識別子付き成果物

- 要件: 配列位置だけで軸・セル・辺・面の意味を復元しない
- 必須実装: 台帳へ実際の識別子を保存し、関係場レコードの対象識別子を台帳と照合する
- 禁止する近道: 件数から識別子を暗黙生成して未知識別子を受け入れる、欠測を0で埋める
- 検証: 任意文字列識別子を持つ小構造を検証し、未知軸参照・未来予測証拠・不正な値区間を拒否する

### 15.5 自己申告を認めない保存検証

- 要件: `passed: true`だけでは成果物を有効としない
- 必須実装: 検証時にマニフェスト、台帳、接続、面境界、質量、対象識別子を再計算する
- 禁止する近道: 生成時の検証JSONを読み、その値をそのまま返す
- 検証: 構造成果物を1バイト変更した否定試験を失敗させ、二重生成の全ファイルバイト一致を確認する

## 16. 簡易実行契約

簡易実行でも正式経路を省略しない。縮小してよいのはG/Kの時点数と関係場レコード数だけである。

| 正式段階 | 簡易実行 |
| --- | --- |
| 既存RF-2生成・検証 | 正式RF-2生成器・検証器をそのまま実行 |
| 固定5軸構造接続 | 正式G2構造生成関数をそのまま実行 |
| RF-6面互換 | 正式RF-6面生成器と全20,000面を比較。縮小しない |
| 構造成果物再検証 | 正式G2検証器をそのまま実行 |
| G/K接頭部 | 正式G/K生成器で4時点を作り、正式G2関数で2時点へ接続 |
| 関係場レコード | 正式G2関数で2レコードを保存・再検証 |

専用Actionsは新規テストファイル全体を実行するため、上記の全経路を通る。

## 17. 試験契約

### 正試験

- G2契約と固定5軸対応表の正本参照
- 構造成果物の二重生成完全一致
- RF-2セル・辺・接続完全一致
- RF-3輸送費用完全一致
- RF-6面完全一致
- `B_1 B_2 = 0`
- 固定値を持たない小構造の検証
- 因果的G/K接頭部と元成果物非変更
- 識別子付き関係場レコードの決定論的保存

### 否定試験

- 予測実装済みという偽契約を拒否
- 動的構造対応済みという偽プロファイルを拒否
- 接続符号反転を拒否
- マニフェスト改ざんを拒否
- G/K来歴不一致を拒否
- 存在しない予測時点を拒否
- 未知の対象識別子を拒否
- 観測成果物への`forecast`証拠混入を拒否

## 18. 停止条件

次の場合、値の推測、既定値補完、条件緩和をせず停止する。

- 必要な既存契約または成果物がない
- 既存契約同士で軸・形状が一致しない
- RF-2/RF-3/RF-6との完全比較が成立しない
- 構造ハッシュまたはマニフェストが一致しない
- 指定されたG/K時点が存在しない
- G/K接頭部が連続していない
- 固定G/K以外の正式対応が必要になる
- 動的構造対応が必要になる
- PR #173またはPR #149の変更が必要になる

## 19. 受け入れ条件

- 新規G2試験が全件通過する
- 既存G/K・RF-1〜RF-10回帰試験が全件通過する
- 構造検証結果がセル・辺・面・接続・費用の完全一致を示す
- 二重生成した構造成果物の全ファイルが同一バイト列になる
- 構造成果物の構造ハッシュとSHA-256マニフェストが再検証できる
- 因果的接頭部の元G/Kファイルハッシュが前後で不変である
- `git diff --check`が通る
- 既存ファイルを変更せず、G2専用ファイルの追加だけである
- 予測精度、汎用生成G/K、動的構造対応を実装済みと主張しない

## 21. 近道監査

| 問い | 判定 |
| --- | --- |
| 最も安い偽実装は何か | 件数と`passed: true`だけを保存する実装 |
| 固定値・空値・`true`・`0`で実計算を置換できるか | できない。全台帳・接続・面・質量を再計算する |
| 生成器が自分の出力を自己認証できるか | できない。再検証と既存RF生成器との比較が必要 |
| 簡易実行が正式分岐を省略できるか | できない。全正式関数を実行する |
| 否定試験はあるか | ある。契約、構造、来歴、改ざん、識別子、証拠状態を壊す |
| 設定が唯一の正本か | はい。固定値は既存凍結契約だけから読む |
| 未実装を妥当そうな既定値で隠せるか | できない。能力不足・未実装・予測なしを明示する |
| 最小の偽実装が試験を通るか | いいえ |
